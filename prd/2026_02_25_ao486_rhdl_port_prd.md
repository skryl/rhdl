# ao486 RHDL Port — Product Requirements Document

**Status:** In Progress
**Date:** 2026-02-25
**Branch:** `claude/plan-ao486-rhdl-port-bnKtw`

---

## Context

The [ao486](https://github.com/MiSTer-devel/ao486_MiSTer) is a cycle-accurate, open-source Intel 486-compatible CPU core originally written in Verilog for the MiSTer FPGA platform. It implements the full x86-32 protected-mode instruction set (118 command types), a 5-stage pipeline, microcode engine, TLB with paging, instruction/data caches, and a PC-compatible SoC with classic peripherals (8259 PIC, 8254 PIT, 8237 DMA, VGA, PS/2, RTC, IDE, floppy).

This PRD covers re-implementing the **CPU core** in RHDL under `examples/ao486/hdl/`, following the established patterns from existing RHDL examples (MOS 6502, Apple II, Game Boy, RISC-V). The reference Verilog lives in the submodule at `examples/ao486/ao486_MiSTer/`.

### Source material scale

| Subsystem | Verilog LOC | Files | Complexity |
|-----------|-------------|-------|------------|
| CPU pipeline (fetch/decode/read/execute/write) | ~10,700 | 28 | Very High |
| Memory subsystem (TLB, caches, prefetch) | ~4,600 | 13 | High |
| Autogenerated modules (microcode/commands) | ~7,100 | 10 | High (generated) |
| Defines & global regs | ~570 | 2 | Low |
| Exception handling | ~540 | 1 | High |
| **CPU core total** | **~24,100** | **56** | — |
| SoC peripherals (PIC, PIT, DMA, VGA, etc.) | ~6,400 | 18 | High |
| Caches (L1 icache, L2) | ~1,040 | 2 | Medium-High |
| System integration | ~850 | 1 | High |

For comparison, the largest existing RHDL example (RISC-V RV32I) is ~8,400 LOC across 32 files. This port will be roughly **3x that scale** and is the most complex RHDL example attempted.

## Goals

1. **Faithful CPU core port**: Re-implement the ao486 CPU pipeline, memory subsystem, exception handling, and microcode engine in RHDL, preserving cycle-accurate behavior against the reference Verilog.
2. **Incremental testability**: Every phase produces runnable tests that verify correctness against the Verilog reference before moving on.
3. **Leverage latest RHDL DSL features**: Use the full DSL surface — `behavior` with `case_select`/`case_of`/`mux`/`local`, `sequential` with reset_values, `state_machine`, `memory` with sync/async ports, `vec`/`input_vec`/`output_vec` for register arrays, `Bundle` for bus interfaces, `parameter` for reusable widths — to keep implementation concise (target 40-60% of the Verilog LOC). Use Ruby metaprogramming to generate dispatch tables from the command definition files instead of manually translating the 7K LOC of autogenerated Verilog.
4. **SoC peripherals as a later scope**: This PRD covers the CPU core only. SoC peripherals (PIC, PIT, DMA, VGA, PS/2, etc.) will be a follow-up PRD once the core is validated.

## Non-Goals

- Porting the MiSTer FPGA platform wrapper (`ao486.sv`, PLL, HPS interface).
- Porting the OPL3 sound module, UART VHDL, or CDDA.
- Verilog-to-RHDL automatic translation tooling.
- Performance parity with FPGA implementation (simulation speed is a non-goal).
- Synthesizable Verilog export (can be explored later, not a requirement).

## DSL Feature Usage Guidelines

The ao486 port should leverage the **latest RHDL DSL features** to minimize boilerplate and keep the RHDL implementation significantly more concise than the reference Verilog. Target roughly **40-60% of the Verilog LOC** (10-14K lines vs 24K) by applying the following features systematically:

### Inline expressions over intermediate wires

Use `local(name, expr, width:)` inside behavior blocks instead of declaring standalone `wire` + separate assignment for one-off intermediates. Reserve explicit `wire` declarations for signals shared across multiple blocks or sub-components.

```ruby
# Prefer:
behavior do
  sum = local(:sum, a + b + c_in, width: 9)
  result <= sum[7..0]
  carry  <= sum[8]
end

# Over separate wire + behavior:
wire :sum, width: 9
# ...then assign in behavior
```

### `case_select` for dispatch tables

The ao486 microcode dispatch (118 CMD_* types) maps naturally to `case_select` hash tables. Use `case_select(selector, { val => expr, ... }, default:)` instead of deeply nested if/elsif chains.

```ruby
behavior do
  result <= case_select(cmd, {
    CMD_ADD => a + b,
    CMD_SUB => a - b,
    CMD_AND => a & b,
    CMD_OR  => a | b,
  }, default: 0)
end
```

### `case_of` for multi-output dispatch

When a single selector controls several output signals, use the multi-output `case_of` block instead of writing parallel case_select trees for each output.

### `mux` over verbose if-else

Use `mux(cond, when_true, when_false)` for two-way selection instead of full if_expr blocks, especially when composable (nested muxes for priority logic).

### `cat` and bit slicing for field packing

x86 is full of packed fields (descriptors, EFLAGS, ModR/M). Use `cat(high, low)` for assembly and `signal[hi..lo]` for extraction instead of manual shift/mask arithmetic.

```ruby
# Descriptor cache packing
descriptor <= cat(base[31..24], flags, limit[19..16], access, base[23..0], limit[15..0])

# EFLAGS field extraction
cf <= eflags[0]
iopl <= eflags[13..12]
```

### `input_vec` / `output_vec` / `vec` for register arrays

The GPR file (8 x 32-bit), segment descriptor caches (8 x 64-bit), and TLB entries are natural fits for `vec` instead of spelling out individual ports.

```ruby
vec :gpr, count: 8, width: 32          # EAX..EDI
vec :seg_cache, count: 8, width: 64    # Descriptor caches
```

### Bundles for repeated bus interfaces

Define a bundle for the memory bus interface (address, data, read, write, ready, valid) and reuse it for prefetch, read, and write stage connections instead of redeclaring 6+ ports per interface.

```ruby
class MemBus < RHDL::Sim::Bundle
  field :addr,  width: 32, direction: :output
  field :data,  width: 32, direction: :inout
  field :read,  width: 1,  direction: :output
  field :write, width: 1,  direction: :output
  field :ready, width: 1,  direction: :input
end
```

### `state_machine` for pipeline/microcode FSMs

The prefetch controller, memory read/write FSMs, and microcode sequencer all have explicit state machines in the Verilog. Use the `state_machine` DSL with declarative states and transitions instead of hand-coded state registers and next-state logic.

### `memory` DSL for caches and FIFO storage

The prefetch FIFO, TLB entry storage, and icache tag/data arrays should use `memory :name, depth:, width:` with `sync_write` / `async_read` ports instead of hand-managing arrays.

### Parameterized components

Use `parameter` for components that vary by width or depth (register file width, cache depth, TLB associativity) so the same component class works for different instantiations.

### Ruby metaprogramming for generated dispatch

The 7,100 LOC of autogenerated Verilog (76 command `.txt` files → `*_commands.v`) should be replaced with Ruby code that reads the command tables and generates `case_select` / `case_of` branches programmatically, rather than manually translating each generated file.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| **Scale**: 24K LOC Verilog → estimated 10-14K LOC RHDL (leveraging DSL features for ~40-60% reduction). Still the largest port by 2x. | High | Phased approach with exit criteria per phase. Each phase is independently valuable. Aggressive use of DSL features (vec, bundles, case_select, memory DSL, metaprogrammed dispatch) keeps LOC manageable. |
| **Autogenerated modules**: 7K LOC of generated Verilog (command dispatch, microcode). | High | Study the generation scripts in `sw/`. Consider writing a Ruby generator that reads the .txt command files and produces RHDL dispatch logic. |
| **Cycle accuracy**: Subtle pipeline timing differences could break real software. | Medium | Use Verilog testbenches from `sim/` as golden reference. Add cycle-count assertions early (Phase 5). |
| **Microcode complexity**: 76 command definition files with complex multi-step sequences. | High | Port incrementally by instruction family. Start with simple (MOV, ADD) and build toward complex (task switch, IRET). |
| **Protected mode corner cases**: Ring transitions, task switches, gate descriptors have intricate interactions. | High | Dedicated phase (8) with thorough spec coverage. Use Intel SDM as authoritative reference alongside Verilog. |
| **RHDL DSL limitations**: Some Verilog patterns (e.g., deeply nested case statements, generate blocks) may not map cleanly. | Medium | Leverage latest DSL features (case_select, vec, bundles, state_machine) and Ruby metaprogramming for generated dispatch. The 118 CMD_* dispatch tables are a natural fit for Ruby hash/case. Identify DSL gaps early in Phase 0-1 and propose extensions if needed. |
| **Test coverage gaps**: Hard to test every x86 edge case. | Medium | Prioritize critical paths (boot, interrupt, protection). Use existing test ROMs and BIOS images from MiSTer community. |
