# Import Guide

This guide documents the current `import` behavior in this repository.

## Current Status

- `rhdl import` is wired through `RHDL::CLI::Tasks::ImportTask` to `RHDL::Import.project(...)`.
- Import internals are implemented under `RHDL::Import::*` (`InputResolver`, frontend adapter/normalizer, mapper, translator, pipeline, writer, checks, report/result types).
- CLI execution writes a report and exits `0`/non-zero based on import/check outcome.

## CLI Surface (`rhdl import`)

```bash
rhdl import [options] [OUT_DIR]
```

Available options:

- `--filelist FILE`
- `--src DIR` (repeatable)
- `--exclude PATTERN` (repeatable)
- `-I, --incdir DIR` (repeatable)
- `-D, --define MACRO[=VALUE]` (repeatable)
- `--dependency-resolution MODE` (`none` | `parent_root_auto_scan`)
- `--compile-unit-filter MODE` (`all` | `modules_only`)
- `--missing-modules MODE` (`fail` | `blackbox_stubs`)
- `--check-profile PROFILE` (`default` | `ao486_trace` | `ao486_trace_ir` | `ao486_component_parity` | `ao486_program_parity`)
- `--top MODULE` (repeatable)
- `--check`
- `--no-check`
- `--check-scope SCOPE`
- `--check-backend BACKEND`
- `--expected-trace FILE`
- `--actual-trace FILE`
- `--expected-trace-cmd CMD`
- `--actual-trace-cmd CMD`
- `--trace-command-cwd DIR`
- `--trace-key KEY` (repeatable)
- `--trace-cycles N`
- `--trace-reference-root DIR`
- `--trace-converted-export-mode MODE` (`component` | `dsl_super`)
- `--vectors N`
- `--seed N`
- `--report FILE`
- `--keep-temp`

`OUT_DIR` is optional; default output is `./rhdl_import`.
Import requires `--filelist FILE` or at least one `--src DIR`.

Execution behavior:

```bash
$ rhdl import --src ./rtl --top top
Import success: converted=..., failed=..., checks_failed=...
Output: ...
Report: ...
```

Exit behavior:

- `0`: success
- non-zero: partial conversion failure, check failure, or tool/internal failure

Tooling prerequisites:

- Frontend ingestion requires `verilator` on `PATH`.
- Differential checks use `iverilog` first with `verilator` fallback.
- Import invokes Verilator JSON-only mode with `-Wno-fatal` so warning-only frontends do not hard-fail.
- Common warnings (for example `MULTITOP`) are preserved in frontend stderr metadata; use `--top` to pin intended tops when needed.

## Input Modes (Implemented in `RHDL::Import::InputResolver`)

The resolver supports two modes with filelist precedence:

1. `filelist` mode
- Selected when `filelist` is provided.
- Parses nested filelists via `-f` / `-F`.
- Supports include/define directives from filelists: `-I`, `-D`, `+incdir+`, `+define+`.
- Resolves file paths relative to each filelist's directory.
- Merges CLI `incdir` and `define` values with filelist values (deduplicated, stable order).

2. `autoscan` mode
- Selected when `filelist` is not provided and at least one `src` directory is provided.
- Recursively finds `*.v` and `*.sv` files under each source root.
- Produces deterministic file ordering (`sort`).
- Uses CLI `incdir` and `define` lists directly (deduplicated where applicable).

Shared behavior:

- `exclude` patterns are applied in both modes using `File.fnmatch?` (supports matching absolute path, relative path, and basename).
- If neither `filelist` nor `src` is supplied, resolver raises `ArgumentError`.
- The resolver returns a normalized contract including:
  - `mode`, `filelist_path`, `source_roots`, `source_files`, `include_dirs`, `defines`, `exclude_patterns`
  - `frontend_input` (`source_files`, `include_dirs`, `defines`)

## Output Layout (Implemented by `ProjectWriter` + `Report`)

When `RHDL::Import::Pipeline` runs, it writes output under `out` in this shape:

```text
<out>/
  lib/
    <project_slug>.rb
    <project_slug>/modules/
      <module_name>.rb
  vendor/source_hdl/
    <original_source_basename>
  rhdl_import.yml
  reports/
    import_report.json
    differential/                  # when checks run
      <top>_differential.json
  tmp/checks/                      # when checks run
    <top>/
```

Notes:

- Module files are emitted only for converted/kept modules.
- Module filenames are sanitized/snake-cased from module names.
- `vendor/source_hdl` stores source copies by basename.
- `rhdl_import.yml` includes `project`, `generated_at`, `modules`, and copied `source_files` basenames.
- `reports/import_report.json` is always generated by the report writer when pipeline runs.
- `reports/import_report.json` includes `blackboxes_generated` and `summary.blackboxes_generated`.

## Partial-Output and Failure Semantics (`Pipeline`)

`Pipeline` is explicitly designed to produce partial output:

- Converted modules can still be written even when some modules fail.
- Failed modules are recorded in report payloads.
- Modules that depend (directly or transitively) on failed modules are pruned and recorded with:
  - `code: failed_dependency`
  - `failed_dependencies: [...]`

Top selection behavior:

- If explicit tops are provided, they must exist or `TopDetector` raises `ArgumentError`.
- Without explicit tops, detected tops are modules with no inbound dependency edges.

Result/report status behavior:

- `success` only if:
  - no failed modules remain after pruning, and
  - all checks are `pass` or `skipped` (or checks are disabled)
- `failure` if:
  - any module conversion/prune failure exists, or
  - any check result is non-passing (for example `fail` or `tool_failure`)

Missing module policy behavior:

- `missing_modules=fail` (default) records unresolved dependencies as `missing_module` failures and prunes dependent modules from output.
- `missing_modules=blackbox_stubs` synthesizes deterministic blackbox module files for unresolved dependencies and records them in `blackboxes_generated`.
- Importer-generated module sources that define `def self.to_verilog` or `def self.to_verilog_generated` are rejected as `forbidden_custom_verilog_export`, and dependent modules are pruned.

## Check Behavior (`Pipeline` + `Checks::*`)

Checks are enabled by default when pipeline runs.

Toggle behavior:

- `no_check` disables checks unconditionally.
- Otherwise, `check` defaults to enabled when not explicitly set.

Profile behavior:

- `check_profile=default` uses vector/waveform differential checks (`iverilog` with `verilator` fallback).
- `check_profile=ao486_trace` uses trace-event comparison and writes artifacts under `reports/trace/`.
- `check_profile=ao486_trace_ir` uses the same trace contract as `ao486_trace`, but compares reference Verilog trace events against converted RHDL IR simulation trace events.
- `check_profile=ao486_program_parity` runs a program-backed tri-simulator harness for top `ao486`:
  - reference Verilog simulation
  - generated Verilog simulation
  - generated IR simulation
  and compares instruction-fetch PC/data sequence plus memory writes/final tracked memory words.
- Trace profile inputs can be provided via API (`expected_trace_events` / `actual_trace_events`), CLI JSON files (`--expected-trace`, `--actual-trace`), or commands that emit JSON (`--expected-trace-cmd`, `--actual-trace-cmd`).
- Trace commands run with deterministic context env vars: `RHDL_IMPORT_TOP`, `RHDL_IMPORT_OUT`, `RHDL_IMPORT_PROJECT_SLUG`, `RHDL_IMPORT_CHECK_PROFILE`, `RHDL_IMPORT_TRACE_KIND`.
- Command templates support `%{top}`, `%{out}`, `%{project_slug}`, and `%{profile}` interpolation.
- `--trace-key` (or API `trace_keys`) limits comparison to selected top-level event keys for deterministic bus/debug-subset checks.
- If trace profile is selected but trace events are not provided and no built-in harness path applies, checks are recorded as `skipped` with reason `no_trace_events`.
- For `check_profile=ao486_trace` with top `ao486`, if no explicit trace inputs are provided, importer auto-runs the built-in ao486 harness for expected (reference) and actual (converted) traces.
- For `check_profile=ao486_trace_ir` with top `ao486`, if no explicit trace inputs are provided, importer auto-runs the built-in ao486 harness for expected (reference) and actual (converted IR) traces.
- For `check_profile=ao486_program_parity`, importer loads a deterministic sample program at reset vector `0xFFFF_FFF0`, executes bounded cycles, and requires observable reset-vector execution (instruction fetches including reset-vector line) while enforcing cross-simulator parity for PC/instruction sequences and tracked memory words.
- `--trace-cycles` controls built-in harness cycle budget (default: `1024`), and `--trace-reference-root` overrides reference RTL root.
- `--trace-converted-export-mode` controls how converted traces are sourced:
  - `component` (default): use generated component `.to_verilog` directly.
  - `dsl_super`: prefer inherited DSL export (`super`) when present; otherwise fall back to component `.to_verilog`.

ao486 deterministic harness workflow:

- `examples/ao486/tools/capture_trace.rb` emits canonical JSON trace events for:
  - `--mode reference` (reference RTL source set),
  - `--mode converted` (Verilog exported from generated import modules under `<out>/lib/<project>/modules/*.rb`).
- `capture_trace.rb` also supports `--mode converted_ir` for converted RHDL IR simulation trace capture.
- `capture_trace.rb` supports `--converted-export-mode component|dsl_super` for converted-mode export control.
- Importer-generated module files do not define custom `to_verilog` methods; converted trace export relies on standard RHDL export paths.
- The harness runs a bounded deterministic Verilator testbench and auto-generates temporary blackbox stubs for unresolved modules from source instantiation signatures.
- Canonical event schema includes:
  - transaction events: `avm_read`, `avm_write`, `io_read`, `io_write`, `interrupt_done`
  - per-cycle sample event: `sample` (`cycle` + top bus/control subset keys)
- Recommended parity command pair:

```bash
--expected-trace-cmd "bundle exec ruby examples/ao486/tools/capture_trace.rb --mode reference --top %{top} --out %{out} --cycles 1024"
--actual-trace-cmd   "bundle exec ruby examples/ao486/tools/capture_trace.rb --mode converted --top %{top} --out %{out} --cycles 1024"
```

IR parity command pair:

```bash
--expected-trace-cmd "bundle exec ruby examples/ao486/tools/capture_trace.rb --mode reference --top %{top} --out %{out} --cycles 1024"
--actual-trace-cmd   "bundle exec ruby examples/ao486/tools/capture_trace.rb --mode converted_ir --top %{top} --out %{out} --cycles 1024"
```

Program parity command:

```bash
rhdl import \
  --src examples/ao486/reference/rtl/ao486 \
  --dependency-resolution none \
  --compile-unit-filter modules_only \
  --missing-modules blackbox_stubs \
  --top ao486 \
  --check-profile ao486_program_parity \
  --trace-cycles 256 \
  examples/ao486/hdl
```

Mismatch triage workflow:

1. Inspect `reports/trace/<top>_trace.json` for `first_mismatch`.
2. Check import report check entry (`trace_sources`, `summary.events_compared`, `summary.first_mismatch`).
3. Re-run with a smaller `--cycles` value to isolate earliest divergence.
4. Use `--trace-key` to narrow comparison to specific top-level event keys when debugging.

Top selection for checks:

- Default: converted detected tops only.
- `check_scope` supports:
  - `all`
  - comma-separated top list
  - array input

If `check_scope` explicitly includes a non-converted top, that top is recorded as `skipped` (`top_not_converted`).

Backend behavior:

- Default commands are built for both backends:
  - Icarus: `iverilog -g2012 -s <top> -o <top>.simv ...`
  - Verilator: `verilator --binary --top-module <top> ...`
- `check_backend` can prefer/force one backend (`iverilog`/`icarus` or `verilator`).
- Runner tries Icarus first and falls back to Verilator on `unavailable` or `tool_failure`.

Per-top statuses:

- `pass` / `fail` from waveform comparison.
- `tool_failure` when backend execution does not return `:ok`.
- `skipped` reasons include:
  - `top_not_converted`
  - `no_source_files`
  - `no_backend_command`

Artifacts:

- Per-top check reports are written to `reports/differential/<normalized_top>_differential.json`.
- Import report summary includes `checks_run` and `checks_failed` counts.

## Reproducibility Knobs

Implemented deterministic controls:

- `vectors` (default `16`) controls vector count.
- `seed` (default `1`) controls deterministic stimulus generation.
- Stimulus generation is stable for the same module signature + `vectors` + `seed`.
- Input vector ordering is deterministic (`cycle` increasing; input ports in deterministic signature order).
- Autoscan source discovery is sorted.
- Mismatch serialization in check reports is sorted by cycle/signal.

Useful practices:

- Pin `--seed` and `--vectors` for repeatable checks.
- Pin `--check-backend` and tool versions (`iverilog`/`verilator`) for cross-machine parity.

Known non-determinism:

- `generated_at` timestamps in generated report/config files.
